Swift Performance Optimisation for Software Ray Tracing

Viktor Chernikov June 12, 2026

How I made a pure Swift path tracer run fast(er).

Some time last year I stumbled on one try! Swift conference talk titled "Graphics like Pixar using Swift" by Michael Petrie a.k.a. Kapsy. The talk was about writing a software ray tracing renderer in Swift. Instantly, I became interested, and the question on my mind was about performance. I knew it was not optimised, but nevertheless, how fast a non-optimised code can run? I grabbed the original repo, and started profiling.

The original project didn't produce any output on the screen. Instead, each rendered frame was converted to a video frame, and the end result was a short smooth video. That was not convenient for quick feedback, so I started adding a way to display that in a window. I suppose the original code could have been truly cross-platform, but I needed a UI, sorry 😅

So, the first test done, and I've got approximately 2.3 seconds per frame on start. Let's see what Time Profiler will show. There's a trick using this tool for high intensity CPU apps: ignore the default call tree, ignore the flame graph; instead click the toggle to invert the call tree and see what comes on the top:

Can you see what's wrong here? In a program that should spend all CPU resources number crunching the functions where the time spent the most are all about managing Swift reference pointers. This is something many programmers take for granted, as unavoidable payment to use the language, similar to garbage collector overhead in other languages. In Swift release and retain calls are atomic and relatively expensive. They are necessary to keep track of reference objects' lifetimes when the references are passed across function boundaries. But we don't have to have them here.

Swift is very convenient to use with value types, where mutability can be safely controlled. So, I looked for some classes (reference types) to eliminate. And there it was, a small data chunk used to store ray hit info:

class HitRecord {
    var dist = Float(0)
    var primRef:Sphere? = nil
    var primType:PrimativeType = .sphere
}

Then I just replaced class keyword with struct. In total, all the changes I had to do to make it compile were these:

// struct instead of class
struct HitRecord {
    var dist = Float(0)
    var primRef:Sphere? = nil
    var primType:PrimativeType = .sphere
}

// added `inout` to pass the HitRecord as mutable...
func traverseSpheres(_ ray: inout Ray, _ hit: inout HitRecord) {

// ... so it can be called like this
traverseSpheres(&ray, &hit)

// and replaced `let` with `var` for mutability
var hit = HitRecord()

The result: 0.96 seconds per rendering frame. That was rather an impressive speedup of 240% for such a tiny change! Let's hunt down remaining classes. Found one – Texture – which is used way less often than HitRecord. But replacing it with a struct still shortened frame render time, to 0.92 seconds per frame. Neat!

So far it was very basic. And here lies the lesson: in Swift you don't have to work harder to get the decent performance right on start; you only have to be mindful of your choices. Use reference types (classes and actors) only when necessary, otherwise stick to value types (structs and enums). There is an important exception to this rule: when the optimiser manages to get rid of unnecessary retain / release calls, reference types become faster due to not incurring copying costs. Copy is cheap, but isn't free. The rough outline of function calling overhead is unretained pointer << value copy << managed pointer. Now let's finally move on to the part where we start adding stuff.

Swift 6.2 introduced a very useful type for performance optimisations: fixed size array called InlineArray. If used in the right place, the benefits should be big. And I found this place in Perlin noise texture generator (the sphere in the centre uses it). The generator uses 2x2x2 three-dimensional array as a temp storage. Let's try replacing it:

// Crazy Swift multi dimensional array syntax.
var c:[[[Float]]] = Array(repeating:
                        Array(repeating:
                        Array(repeating: 0.0,
                            count: 2),
                            count: 2),
                            count: 2)

// ... becomes ...
var c: InlineArray<8, Float> = .init(repeating: 0)

Of course, going one dimensional from three complicates indexing access a little bit. But now rendering takes 0.74 seconds per frame! And this is the perfect use case for InlineArray: local temporary storage. Cases like that are not that common, still, it's good to know about InlineArray and keep it ready.

At this point the inverted call tree started to look like this:

The random generator started climbing higher. Random numbers are essential part of path tracing, and the generator is called very often. But can we make it faster? Switching drand48 to standard Swift generator expectedly tanked the performance, so there was a reason to use drand48. I started googling, which led me to a in-depth discussion on Swift forums that gave me a faster (and good enough for this case) option: Wyrand, that dropped rendering time to 0.70 seconds per frame. Here's its complete implementation: 

struct Wyrand: RandomNumberGenerator {
    private var state: UInt64

    init(seed: UInt64 = mach_absolute_time()) {
        state = seed
    }

    mutating func next() -> UInt64 {
        state &+= 0x2d358dccaa6c78a5
        let mul = state.multipliedFullWidth(by: state ^ 0x8bb84b93962eacc9)
        return mul.low ^ mul.high
    }
}

After that point I cleaned up the code a bit, getting to 0.68 seconds per frame. The improvement comes from stricter typing, that allowed optimiser to do more magic behind the scenes. But now the big change finally stood before me: you see, the ray tracer is single threaded. And while sometimes it is easy to parallelise computational work, this code base required some deeper changes.

I'm going to be running parallel tasks. The tasks have to access scene data, which must be immutable, otherwise they will be stuck awaiting each other. No biggie, the scene data is immutable, that should work. After changing the code in several steps I ended up with the following parallel rendering loop:

await withTaskGroup(of: (Int, [V3]).self) { group in
            for j in 0..<height {
                // Each row is running in a separate task
                group.addTask {
                    var row: [V3] = []
                    // Do the rendering
                }
            }
            // Results are stored on completion as (j, [...])
            // where j is row number and array contains colours
            for await result in group {
                results.append(result)
            }
}

But the performance was even worse than before! Time Profiler revealed the issue: the dreaded retain / release calls were back. Where did they come from, if there was no reference types? They came from the value type storage types, like Array. Despite being a struct on the surface, inside the data is stored on the heap with a reference to it. And once I moved access to the scene array into task closures, the optimiser could no longer guarantee the lifetime of the array to outlast the work done in tasks. Hence, reference counting.

Thankfully, Swift 6.2 has another tool to help us:

let spheres: ContiguousArray<Sphere>
// before
spheres.traverseSpheres(ray, &hit)
// after
spheres.span.traverseSpheres(ray, &hit)

Span is a non-owning pointer to the memory window, acting as an Array. Exactly what was needed here to prevent compiler from trying to retain the spheres array, and much more preferable than writing anything starting from "unsafe". With this change the rendering time has fallen to 68 milliseconds per frame, or 10 times faster than single threaded version, or 33 times faster than what we started with. But, of course, that part is very hardware dependent.

Is it the maximum what can be squeezed out? Of course not, but it is good enough for me. Every optimisation I did is not some esoteric voodoo knowledge passed down in secret whispers; it is all simple things. And this is what I hope more programmers would remember: often your code can run much faster, and there's no reason not to make the right choices on the start of your project. 

Thank you.
